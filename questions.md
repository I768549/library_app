## 1) Коротко про проект (готовый устный ответ на 40-60 секунд)

Проект library_app это desktop-приложение библиотеки на Python с интерфейсом Tkinter, базой PostgreSQL и ORM SQLAlchemy.

Архитектура разделена на слои:
- GUI: вкладки, виджеты, взаимодействие с пользователем.
- Repository: бизнес-операции и запросы к БД.
- Models: ORM-модели и связи таблиц.
- Workers: фоновые потоки, чтобы UI не зависал.
- Services: интеграция с локальной LLM (Ollama).

Ключевая идея: тяжелые операции (БД/сеть) выполняются в фоне, а UI обновляется только из главного потока Tkinter через механизм очереди и root.after.

## 2) Карта проекта (что где лежит)

- main.py: точка входа, init_db, seed_database, запуск GUI.

## 4) Построчный разбор workers/db_worker.py (самый важный для защиты)

### Файл целиком

```python
import threading
import queue

class ThreadWorker:
    """Запускає функцію у фоновому потоці та повертає результат у GUI-потік
    через threading.Queue + root.after().
    Кожен виклик run() створює власну чергу
    """

    def __init__(self, root):
        self.root = root

    def run(self, func, *args, on_start=None, on_done=None, on_error=None):
        if on_start:
            on_start()

        q = queue.Queue()`

        def target():
            try:
                result = func(*args)
                q.put(("done", result))
            except Exception as exc:
                q.put(("error", exc))

        threading.Thread(target=target, daemon=True).start()
        self.root.after(100, lambda: self._check(q, on_done, on_error))

    def _check(self, q, on_done, on_error):
        try:
            kind, data = q.get_nowait()
        except queue.Empty:
            self.root.after(100, lambda: self._check(q, on_done, on_error))
            return

        if kind == "done":
            if on_done:
                on_done(data)
        else:
            if on_error:
                on_error(data)
```

### Разбор по строкам и смыслу

1. `import threading` и `import queue`
- threading нужен для фонового потока.
- queue нужен для безопасной передачи результата между потоками.

2. `class ThreadWorker`
- Обертка для повторно используемого шаблона: выполнить задачу вне GUI и вернуть результат в GUI.

3. `__init__(self, root)`
- root сохраняется, чтобы вызывать `root.after(...)` и исполнять callback в main thread Tkinter.

4. `run(func, *args, on_start, on_done, on_error)`
- Принимает функцию и аргументы.
- Принимает три callback-а:
  - on_start: старт операции (показать progress bar).
  - on_done: успех.
  - on_error: ошибка.

5. `if on_start: on_start()`
- Запускается сразу в GUI-потоке, до старта фоновой работы.

6. `q = queue.Queue()`
- Важно: очередь создается на каждый запуск run(), чтобы задачи не смешивались.

7. Внутренняя `target()`
- Это код, который реально выполняется в фоне.
- Если задача прошла успешно, кладем `(done, result)`.
- Если упала, кладем `(error, exc)`.

8. `threading.Thread(..., daemon=True).start()`
- Стартуем отдельный поток.
- `daemon=True` значит поток не блокирует завершение приложения.

9. `self.root.after(100, ...)`
- Запускаем первый poll очереди через 100 мс.
- Tkinter-виджеты изменяем только из GUI thread.

10. `_check(...)` с `q.get_nowait()`
- Неблокирующая попытка получить результат.
- Если очередь пустая: планируем новую проверку через after.

11. Ветвление по `kind`
- done -> `on_done(data)`
- error -> `on_error(data)`

### Почему это корректно для Tkinter

- Tkinter не потокобезопасен.
- Нельзя менять виджеты из worker thread.
- Поэтому поток делает только вычисления/IO и пишет в queue.
- А GUI обновляется через after в главном потоке.

## 5) Построчный разбор chat_tab.py (ключевые куски)

### 5.1 Инициализация состояния

```python
self.queue: queue.Queue = queue.Queue()
self.history: list[dict] = []
self.streaming = False
self._assistant_buf = ""
```

Что это значит:
- `queue` хранит события от фонового запроса: chunk/done/error.
- `history` хранит историю диалога для контекста модели.
- `streaming` блокирует повторную отправку во время ответа.
- `_assistant_buf` собирает полный текст ответа из чанков.

### 5.2 Верхняя панель (модель и URL)

```python
ttk.Label(
    top,
    text=f"Модель: {recommender.DEFAULT_MODEL}    {recommender.OLLAMA_URL}",
).pack(side="left")
```

Что сказать на защите:
- Пользователь видит, какая модель и какой endpoint реально используются.
- Это удобно для диагностики, когда модель/сервер не совпадают.

### 5.3 Асинхронный warmup

```python
def _warmup_model_async(self):
    def worker():
        try:
            recommender.warmup_model()
        except Exception:
            return
    threading.Thread(target=worker, daemon=True).start()
```

Смысл:
- После старта приложения модель прогревается в фоне.
- Первый реальный вопрос становится быстрее.
- Ошибки warmup игнорируются, чтобы не ломать UI.

### 5.4 Отправка сообщения

```python
history_snapshot = list(self.history)

def worker():
    try:
        def on_chunk(chunk: str):
            self.queue.put(("chunk", chunk))
        recommender.chat_stream(history_snapshot, on_chunk)
        self.queue.put(("done", None))
    except Exception as exc:
        self.queue.put(("error", exc))
```

Почему snapshot важен:
- История, с которой работает поток, не должна меняться в процессе запроса.

### 5.5 Poll очереди и стриминг в UI

```python
kind, data = self.queue.get_nowait()
if kind == "chunk":
    self._append(data)
    self._assistant_buf += data
elif kind == "done":
    self.history.append({"role": "assistant", "content": self._assistant_buf})
```

Смысл:
- chunk выводится сразу (потоковый UX).
- done фиксирует итог в истории.
- error показывает messagebox и завершает сессию.

### 5.6 Почему тут root.after(50/100)

- Когда данные идут активно, polling чаще (50 мс).
- Когда очередь пуста, polling реже (100 мс), чтобы меньше грузить CPU.

## 6) Что происходит в services/recommender.py

### Основные идеи

1. Формируется системный промпт + каталог книг из БД.
2. Запрос отправляется в Ollama на `http://localhost:11434/api/chat`.
3. Используется `stream=True`, поэтому ответ читается построчно.
4. Каждый кусок (`content`) передается через callback on_chunk.
5. Используется `keep_alive="30m"`, чтобы модель не выгружалась слишком быстро.
6. Есть `warmup_model(...)` с коротким запросом для прогрева.

### Почему первый вопрос медленнее

- Первый запрос делает cold start (загрузка модели в память).
- Warmup уменьшает этот эффект.

### Что спросит препод

Вопрос: почему timeout в warmup может быть маленьким?
Ответ: warmup best-effort. Если не успел, основной чат все равно работает. Это оптимизация, а не обязательный путь.

## 7) База данных и бизнес-правила (важно)

### 7.1 Связи моделей

- Book <-> Author: many-to-many через `book_author`.
- Book <-> Genre: many-to-many через `book_genre`.
- Loan -> Book и Loan -> Reader: many-to-one.
- ISBN и reader.email имеют unique constraint.

### 7.2 Правила, которые мы отлаживали в диалоге

1. Дубликат email у reader
- Проблема: падение `UniqueViolation`.
- Решение: pre-check + обработка `IntegrityError` с дружелюбным `ValueError`.

2. Detached instance после создания reader
- Проблема: доступ к `reader.id` после закрытия session.
- Решение: `session.flush()` и сохранение id до commit/выхода.

3. Удаление книги после возврата займа
- Проблема: возврат не удаляет loan-строку, FK мешает удалению книги.
- Решение: запрет удаления при активном займе и очистка закрытых loan перед удалением книги.

## 8) Каверзные вопросы и готовые ответы

### 8.1 Потоки vs async

Вопрос: у вас асинхронное программирование или многопоточность?

Ответ:
- Технически это многопоточность (`threading.Thread`).
- Поведение для пользователя асинхронное, потому что UI не блокируется.
- Это не `asyncio`-корутины, а thread + queue + after.

### 8.2 Почему не asyncio?

Ответ:
- Tkinter имеет свой event loop (`mainloop`).
- С `asyncio` можно, но интеграция сложнее и менее очевидна для учебного проекта.
- Для блокирующих IO (БД/HTTP) тут практичнее потоковый шаблон.

### 8.3 Почему не просто вызвать callback из потока?

Ответ:
- Потому что callback обычно трогает Tkinter-виджеты.
- Tkinter не потокобезопасен, это риск краша/гонки.
- Поэтому данные передаются через queue, а callback вызывается из GUI thread.

### 8.4 Почему daemon=True?

Ответ:
- Чтобы фоновые потоки не удерживали процесс при закрытии окна.
- Для короткоживущих операций UI это практично.

### 8.5 Какие минусы текущего подхода?

Ответ:
- На каждую задачу создается новый поток.
- Нет отмены задач и централизованного пула.
- Polling через after это компромисс, а не идеал.

### 8.6 Что можно улучшить?

Ответ:
- Перейти на ThreadPoolExecutor.
- Добавить токен отмены задач.
- Добавить таймауты и ретраи в worker-уровне.
- Логирование в отдельный модуль.

### 8.7 Что такое race condition в вашем контексте?

Ответ:
- Пример: два запроса одновременно создают reader с одинаковым email.
- Даже с pre-check оба могут пройти проверку.
- Поэтому нужен финальный guard на уровне БД (unique + IntegrityError).

### 8.8 Где у вас потокобезопасность обеспечена явно?

Ответ:
- В использовании `queue.Queue()` между потоками.
- И в том, что UI обновления идут через `root.after(...)`.

## 9) Tkinter теория, которую полезно сказать на защите

1. `mainloop()` это цикл обработки GUI-событий.
2. Любая долгая операция в main thread блокирует интерфейс.
3. `after(ms, callback)` ставит callback в очередь GUI-цикла.
4. `StringVar.trace_add(...)` реагирует на изменения переменной (например, live-search).
5. `Toplevel + transient + grab_set` делает модальное окно.
6. Для `Text/ScrolledText` в режиме disabled нужно временно включать normal для записи.
7. `Treeview` не хранит типы, сортировка часто делается вручную.

## 10) Блиц-ответы (30-60 секунд каждый)

### 10.1 Почему у вас не виснет UI?
Потому что тяжелые операции выполняются в отдельных потоках, а в GUI-поток возвращается только готовый результат через queue и after.

### 10.2 Почему chat отвечает кусками?
Потому что запрос к Ollama идет с `stream=True`, а каждый chunk сразу выводится в виджет.

### 10.3 Почему first response медленнее?
Cold start модели. Мы добавили warmup и keep_alive.

### 10.4 Что такое активный займ?
Займ, у которого `returned_at is None`.

### 10.5 Почему нельзя удалять книгу с активным займом?
Бизнес-правило + защита целостности данных (FK).

### 10.6 Почему вы обрабатываете IntegrityError отдельно?
Чтобы вернуть понятную доменную ошибку в UI, а не сырой SQL-стектрейс.

### 10.7 Чем отличается return loan от delete loan?
Return заполняет `returned_at` (история сохраняется), delete удаляет строку полностью.

### 10.8 Где проявляется separation of concerns?
GUI только управляет состоянием и отображением, repository отвечает за данные и правила.

## 11) Мини-конспект «если препод давит в теорию»

### Синхронность / асинхронность / параллелизм

- Синхронный код: шаги идут последовательно, текущий шаг блокирует следующий.
- Асинхронный код: можно не блокировать главный поток ожиданием IO.
- Конкурентность: несколько задач в прогрессе в одно время.
- Параллелизм: задачи реально исполняются одновременно на разных ядрах.

В проекте:
- Используется конкурентность через потоки.
- UI не блокируется, но это не coroutine-asyncio архитектура.

### GIL кратко

- В CPython GIL ограничивает одновременное выполнение Python-байткода в потоках.
- Для IO-bound задач (сеть/БД), как у нас, потоки полезны и работают хорошо.
- Для CPU-bound задач лучше multiprocessing.

## 15) Вопросы по запуску на Debian (тоже могут спросить)

### 15.1 Что нужно скачать для проекта?

Минимально:
- Python-зависимости из requirements.txt
- PostgreSQL сервер + клиент
- Ollama + модель llama3.2

Типовой порядок:

```bash
sudo apt update
sudo apt install -y postgresql postgresql-client
pip install -r requirements.txt

# Ollama
curl -fsSL https://ollama.com/install.sh | sh
ollama pull llama3.2
```

### 15.2 Почему psql ругался на cluster/client?

Потому что был установлен не полный набор пакетов клиента/сервера PostgreSQL.
Нужен как минимум `postgresql` и `postgresql-client` (иногда конкретная версия `postgresql-client-16`).

### 15.3 Что значит команда вида `curl ... | sh`?

Это передача текста из stdout `curl` напрямую в shell `sh` для немедленного выполнения.

Плюс:
- Быстро ставит инструмент.

Минус (важно для теории безопасности):
- Ты выполняешь удаленный скрипт без предварительного аудита.

Безопаснее:
- скачать скрипт в файл,
- открыть и проверить содержимое,
- только потом запускать.

### 15.4 Как проверить, что все сервисы живы?

PostgreSQL:

```bash
sudo systemctl status postgresql
psql -h localhost -U <user> -d <db>
```

Ollama:

```bash
ollama list
curl http://localhost:11434/api/generate -d '{"model":"llama3.2","prompt":"hi","stream":false}'
```

## 16) История реальных багов и как ты их исправил (сильный аргумент на защите)

1. Duplicate email у читателя
- Симптом: `UniqueViolation` с длинным SQL-текстом.
- Исправление: pre-check email + `IntegrityError` fallback + понятный `ValueError`.

2. Detached instance при создании reader
- Симптом: `Instance ... is not bound to a Session`.
- Исправление: `session.flush()`, сохранить `reader_id`, вернуть id до закрытия сессии.

3. Нельзя удалить книгу после возврата
- Симптом: FK-конфликт из-за исторических loan-строк.
- Исправление: блокируем удаление только при активных займах, закрытые loan удаляем перед delete book.

4. Первый ответ ассистента медленный
- Симптом: первый вопрос заметно дольше остальных.
- Исправление: warmup в фоне + keep_alive.

Этот блок можно использовать как «пруф», что ты понимаешь не только happy path, но и диагностику production-проблем.
